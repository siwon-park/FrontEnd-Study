# 07_제어 컴포넌트

> Controlled Vs. Uncontrolled Component

HTML에서 `<input>` 태그는 원래 자기 자신의 상태를 스스로 가지고 있는 요소이다. 사용자가 값을 입력하면 브라우저(DOM)이 알아서 그 값을 기억하고 있다.

그러나 React의 등장 이후 이러한 개념에 변화가 있었는데, 바로 React가 "UI는 상태(state)의 함수다"라는 철학을 가지고 있기 때문이다. 화면에 보이는 모든 것이 어떤 state에서 파생되어야 한다는 것이다.

그런데 input의 값은 DOM이 자체적으로 관리하니까 React 입장에서는 "내가 모르는 상태가 화면 어딘가에 숨어있는" 상황에 놓여지게 된다.

여기서,

- React state와 input의 값을 강제로 동기화시켜서 React가 제어하게 만들기
- DOM에 관리를 맡기고, React는 필요할 때만 참조하기

라는 두 가지 선택지가 생기게 되었다.

## 1. 제어 컴포넌트

React state가 화면의 값을 제어하는 방식.

```typescript
const Controlled = () => {
    const [value, setValue] = useState('');
    
    return (
        <input
        	value={value}
			onChange={(e) => setValue(e.target.value)}
        />
    );
}
```

- 사용자가 실제 타이핑을 해도 input이 자기 스스로 값을 바꾸는 것이 아니다.
- 과정
  - 사용자가 키를 입력한다.
  - onChange 이벤트 발생 => setValue(새값) 호출
  - setState로 인해 상태변화 감지, React가 리렌더링
  - 리렌더링된 input 안의 value가 setState의 값을 기준으로 다시 그려짐
  - 화면 상에서는 마치 사용자가 입력한 그대로 보이는 것과 같은 현상
- 겉보기에는 자연스럽게 값이 바뀌는 것 같지만, 실제로는 매 글자마다 "지우고 화면을 다시 그리는" 행위를 반복하는 것이다.
- 유일한 진실의 원천(single source of truth);
  - input 값을 강제로 바꾸더라도 화면이 바뀌지 않는다.
  - 오로지 state에 의해서 값이 제어된다. 

<br>

## 2. 비제어 컴포넌트

DOM이 값을 제어하는 형식.

```typescript
const Uncontrolled = () => {
  const inputRef = useRef(null);

  const handleSubmit = () => {
    alert(inputRef.current.value);
  };

  return (
    <>
      <input ref={inputRef} defaultValue="초기값" />
      <button onClick={handleSubmit}>제출</button>
    </>
  );
}
```

- React가 타이핑 한 글자 한 글자에 전혀 관련하지 않는다.
- DOM이 알아서 값을 가지고 있다가, React는 필요할 때만 `ref`라는 통로를 이용해서 DOM에서 값을 참조해 오는 것이다.
- 과정
  - input의 기본값이 defaultValue에 의해 "초기값"이라고 세팅됨.
  - 타이핑 => DOM이 값을 계속 저장하고 있음 (React 관여 X)
  - 버튼 클릭 => handleSubmit 메서드 호출
  - ref를 통해 DOM상 정의해놓은 요소 참조(inputRef.current) => input에 입력된 값(inputRef.current.value) 참조

<br>

## 3. 정리

### 1) 언제 써야하는가?

#### (1) controlled component

- 입력하는 즉시 다른 UI에 반영해야 할 때
- 입력값에 대해 조건부 검증이 필요할 때 (예 - 특정 글자수 이상 입력해야 버튼 활성화 등)
- 여러 입력 필드가 서로 연동될 때 (예- 비밀번호와 비밀번호 확인 입력의 일치 체크)
- 입력값 자체를 가공해야할 때 (입력 시 즉시 대문자 변환, 숫자만 허용 등)

#### (2) uncontrolled component

- 단순히 제출할 때 값만 읽으면 되는 폼
- 실시간 반응이 불필요한 경우
- input 타입이 file인 경우 (보안상의 이유로 브라우저가 강제로 값을 절대 코드로 설정 불가능하게 해둠)
- 부모로부터 props로 값을 받은 이후, 해당 요소의 변화에 대해 다시 그릴 필요가 없는 경우

<br>

### 2) 흔히 하는 실수

input에 value state 세팅 후 변경에 대한 함수가 없는 경우

```typescript
// 위험한 패턴: value는 줬는데 onChange가 없음
<input value={value} />
```

- React 입장에서는 value라는 상태가 있어 이를 controlled라고 만들려는 건 줄 알음
- 그러나 value의 변경을 감지할 수 있는 onChange가 없으니 사용자가 값을 입력해도 바뀌지 않는다는 취지의 경고를 발생시킨다. (`Warning: You provided a value prop to a form field without an onChange handler`)
